Data Catalog: A metadata store containing table definitions, job definitions, and other control information for your ETL workflows.
Crawlers: Programs that connect to data sources, infer data schemas, and create metadata table definitions in the Data Catalog.
ETL Jobs: The business logic to extract data from sources, transform it using Apache Spark scripts, and load it into targets.
Triggers: Mechanisms to initiate job runs based on schedules or events.

The typical workflow involves:
Define data sources and targets in the Data Catalog.
Use Crawlers to populate the Data Catalog with table metadata from data sources.
Define ETL jobs with transformation scripts to move and process data.
Run jobs on-demand or based on triggers.
Monitor job performance using dashboards.

You are charged an hourly rate based on the number of Data Processing Units (or DPUs) used to run your ETL job. A single Data Processing Unit (DPU) is also referred to as a worker.

from awsglue.context import GlueContext
from awsglue.job import Job
from pyspark.context import SparkContext
sc = SparkContext()
glueContext = GlueContext(sc)
spark = glueContext.spark_session
job = Job(glueContext)  # Initialize bookmark service
job.init(args['JOB_NAME'], args)
# S3 source with bookmark (tracks new files via transformation_ctx)
datasource0 = glueContext.create_dynamic_frame.from_catalog(
    database="hr",
    table_name="emp",
    transformation_ctx="datasource0"  # Enables bookmark for this step -- must if we want bookmarking
)
# Apply mapping (bookmark tracks transformed state)
applymapping1 = ApplyMapping.apply(frame=datasource0, transformation_ctx="applymapping1")
# Write to S3 sink (bookmark updates on commit)
datasink2 = glueContext.write_dynamic_frame.from_options(
    frame=applymapping1,
    connection_type="s3",
    connection_options={"path": "s3://hr/employees/"},
    format="parquet",
    transformation_ctx="datasink2"
)
job.commit()  # Persists bookmark state

 | Action                        | Effect                                         | Use Case                                             |
| ----------------------------- | ---------------------------------------------- | ---------------------------------------------------- |

| Case | Bookmark | Overwrite | Result                |
| ---- | -------- | --------- | --------------------- |
| A    | ENABLED  | OVERWRITE | ❌ Partial data wipe   |
| B    | DISABLED | OVERWRITE | ✅ Correct full load   |
| C    | ENABLED  | APPEND    | ✅ Incremental load    |
| D    | DISABLED | APPEND    | ❌ Duplicate explosion |

>>>>>>>>>>>>>>>>> AWS lakehouse formation

AWS Lake Formation is a centralized data governance and access-control service for data lakes on AWS

It does NOT store data
It does NOT process data
It controls who can see and do what with data stored in S3 and cataloged in Glue.

Without Lake Formation
Permissions are controlled using:
S3 bucket policies
IAM policies

Problems:
Coarse-grained (bucket-level)
Hard to manage at scale
No column-level security
No row-level filters
Security logic duplicated everywhere

With Lake Formation
You get:
Database / table / column-level permissions
Central governance
One-time permission definition
Works across Glue, Athena, Redshift Spectrum, EMR


“Glue Catalog is metadata; Lake Formation enforces who can use that metadata.”

Lake Formation + glue job
Glue Job Execution Roles
Must be explicitly granted permissions in Lake Formation:
READ on Bronze
WRITE on Silver/Gold

If not:
Glue job fails with permission error
Even if IAM allows S3 access

📌 Key concept:
“Lake Formation permissions override IAM for registered locations.”

Column-level masking on sensitive fields
Safe concurrent access

Athena / Redshift / Glue all used

Overkill When:

--Blocks insertions of employees under 25, enforcing hiring policy at the database level.
CREATE TRIGGER check_employee_age
BEFORE INSERT ON employees
FOR EACH ROW
BEGIN
    IF NEW.age < 25 THEN
        SIGNAL SQLSTATE '45000' 
        SET MESSAGE_TEXT = 'Employee age must be at least 25';
    END IF;
END;

--Logs salary changes to an audit table whenever an employee record updates
CREATE TRIGGER after_employee_update
AFTER UPDATE ON employees
FOR EACH ROW
BEGIN
    INSERT INTO employee_audit (employee_id, old_salary, new_salary, changed_date)
    VALUES (OLD.employee_id, OLD.salary, NEW.salary, NOW());
END;

NTILE() is a very helpful window function. It helps you to identify what percentile (or quartile, or any other subdivision) a given row falls into.

This means that if you have 100 rows and you want to create 4 quartiles based on a specified value field you can do so easily and see how many rows fall into each quartile.

Let’s see an example. In the query below, we have specified that we want to create four quartiles based on order amount. We then want to see how many orders fall into each quartile.
 
SELECT order_id,order_date,customer_name,city, order_amount,
NTILE(4) OVER(ORDER BY order_amount) [row_number]
FROM [dbo].[Orders]


Here is our example, 
we have total 10 rows and 4 tiles are specified in the query so number of rows in each tile will be 2.5 (10/4). As number of rows should be whole number, not a decimal. SQL engine will assign 3 rows for first two groups and 2 rows for remaining two groups.
